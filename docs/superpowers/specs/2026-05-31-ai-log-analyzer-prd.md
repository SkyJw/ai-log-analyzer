# AI Log Analyzer 需求文档

## 1. 背景

板级软件团队负责通信板卡的底软开发，工作范围包括 bootloader 中的板级适配、部分 Linux OS 驱动、Linux 用户态板级驱动等。日常版本开发和问题定位过程中，上层产品软件团队经常带着日志和问题来找板级团队协助分析。

这类问题通常有几个特点：

- 日志分散在 bootloader、kernel、用户态服务、板级驱动等多个阶段。
- 一份日志包中可能包含最近多次启动记录。
- 同一次启动的不同阶段日志可能分散在不同文件中。
- 有些日志只记录当前启动，有些日志会保留最近多次启动历史。
- 上层软件提出的问题通常是业务化描述，不会直接问底层日志细节。
- 问题定位依赖板级工程师经验，人工阅读日志和组织回复耗时较多。

因此，本产品希望利用内网私有化大模型能力，把板级工程师的分析方法沉淀到系统中，减少重复日志分析和问题答复工作。

## 2. 产品定位

AI Log Analyzer 是一个面向通信板卡的 **启动过程重建与 AI 诊断系统**。

它不是通用日志总结工具，也不是简单的相似案例搜索器。它的核心价值是：

- 从分散日志中重建最近多次启动过程。
- 按照板级工程师沉淀的诊断模板主动分析日志。
- 围绕用户本次提出的问题进行动态回答。
- 将工程师认可的有效诊断结果沉淀为本地问答知识集。
- 在后续问题定位中复用已审核的经验案例。

## 3. MVP 目标

MVP 需要完成以下目标：

- 支持上传多次启动日志包，也支持直接上传单次启动小压缩包。
- 只分析已配置白名单且具备解析能力的日志文件。
- 基于压缩包结构、日志白名单和日志范围策略重建最近 N 次启动过程。
- 基于诊断模板对当前日志做主动诊断。
- 基于启动重建结果、主动诊断结果和日志证据，动态回答用户问题。
- 支持围绕本次分析结果继续追问。
- 支持后台查看每次诊断任务和诊断结果。
- 支持工程师对有效诊断结果点赞、编辑摘要并确认入库。
- 将确认后的结果加入本地问答知识集，供后续诊断引用。
- 第一版接入 MiniMax M2.7 API，同时保留模型可插拔能力。
- 满足内网私有化部署要求，日志不出公司受控环境。

## 4. MVP 不做什么

第一版明确不做以下能力：

- 不做复杂权限隔离。
- 不做自动远程登录板卡执行命令。
- 不替代完整问题单流转系统。
- 不默认分析压缩包中的所有文件。
- 不让 AI 自动修改并生效诊断模板。
- 不做网页化诊断模板编辑器。
- 不做完整源码静态分析平台。

## 5. 用户

MVP 面向两类用户：

- 板级驱动工程师：负责问题定位、诊断模板维护、有效案例审核入库。
- 上层产品软件工程师：上传日志和问题，查看分析结果，继续追问。

第一版两类用户看到的主分析视图保持一致，不做复杂视图差异。

## 6. 总体流程

```text
用户问题 + 日志压缩包
  -> 压缩包安全检查与解压
  -> 输入包类型识别
  -> 启动快照识别
  -> 白名单日志匹配
  -> 日志解析与事实抽取
  -> 第一层：启动过程重建
  -> 第二层：诊断模板主动分析
  -> 第三层：用户问题动态回答
  -> 报告展示与追问
  -> 工程师后台审核
  -> 有效结果确认入库
  -> 后续诊断引用本地知识集
```

## 7. 三层分析架构

### 7.1 第一层：启动过程重建

启动过程重建是固定流程，不依赖用户具体问什么。

这一层回答：

- 这份日志包里有多少次启动快照？
- 每次启动分别发生了什么？
- 每次启动是否进入 bootloader、kernel、userspace、板级服务等阶段？
- 每次启动最终是成功、复位、卡住、部分成功还是无法判断？
- 复位或卡住大概率发生在哪个阶段？
- 每个判断由哪些文件和日志片段支撑？
- 哪些日志缺失、无法识别或无法归属？

### 7.2 第二层：诊断模板主动分析

诊断模板主动分析也是固定流程。它基于已有诊断模板，对当前日志做类似“体检”的主动分析。

这一层回答：

- 当前日志中暴露了哪些已知异常模式？
- 哪些启动会话受影响？
- 命中了哪些诊断模板？
- 每个诊断模板给出了什么发现？
- 有哪些直接日志证据？
- 有哪些疑似根因和下一步排查建议？

诊断模板是人工经验的主要载体。问题单、代码提交记录、人工典型案例、已入库问答案例都是辅助证据，不能替代当前日志中的直接证据。

### 7.3 第三层：用户问题动态回答

用户问题动态回答不是固定模板，也不引入额外的问题翻译模板。

这一层围绕用户本次提出的具体问题，动态决定需要引用哪些信息、补充哪些分析，并组织最终回答。

它可以使用：

- 第一层启动过程重建结果。
- 第二层诊断模板主动分析结果。
- 白名单日志原文片段。
- 文件清单和忽略文件信息。
- 本地已审核问答知识集。
- 内部问题单、代码提交记录、人工典型案例等辅助证据。
- 用户原始问题和后续追问。

这一层回答：

- 当前日志能否解释用户的问题？
- 哪些结论有直接证据支撑？
- 哪些判断只是推断？
- 当前证据不足以回答哪些点？
- 还需要补充哪些日志、版本信息或现场操作信息？
- 该问题更可能属于底软、上层软件、版本配套、配置、硬件还是证据不足？

示例：

用户问：“我做了一次包下载升级，升级后板子没有从预期启动区域启动，为什么？”

第三层不会依赖固定问题模板，而是动态查看：

- 最近几次启动实际从哪个区域启动。
- 主动诊断中是否发现启动区选择、升级回退、镜像校验、复位等异常。
- bootloader 日志、升级日志、环境变量、启动参数中是否有直接证据。
- 当前日志能否证明未从预期区域启动的原因。
- 如果不能证明，还需要补充哪些信息。

## 8. 日志输入要求

### 8.1 上传模式

上传入口需要支持两种输入：

- 多次启动日志包：外层大压缩包内包含多个带编号的小压缩包。
- 单次启动日志包：用户直接上传某一次启动的小压缩包。

系统应自动识别类型，不要求用户手工选择。

识别规则：

- 如果压缩包内存在多个符合编号规则的小压缩包，则识别为多次启动日志包。
- 如果压缩包内直接包含白名单日志，则识别为单次启动日志包。
- 如果两者混合，优先按多次启动日志包处理，根目录日志作为采集级辅助文件。

### 8.2 多次启动日志包

外层大压缩包中包含多个小压缩包。小压缩包尾缀编号表示启动快照顺序：

- `0`：最近一次启动。
- `1`：上一次启动。
- `2`：上上次启动。
- 以此类推。

系统根据编号建立启动快照：

```text
BootSnapshot
- snapshot_index
- display_name
- archive_path
- extracted_files
```

### 8.3 单次启动日志包

如果用户直接上传一个小压缩包，系统将它视为 `snapshot_index = 0`，即当前上传的这一次启动。

如果小包内某些日志本身保留了最近多次启动历史，系统仍然按照日志范围策略尝试切分和归属。但报告主视角是本次上传的启动。

## 9. 日志白名单与范围策略

MVP 不全量分析压缩包中的所有文件，而是通过白名单控制可分析文件。

每个文件会被分类为：

- `analyzed`：命中白名单，且有对应解析能力。
- `recognized_not_analyzed`：能识别类型，但当前未启用分析。
- `ignored`：未知、无关、二进制、过大、路径不安全或不支持格式。

报告必须展示：

- 本次分析使用了哪些文件。
- 哪些文件被识别但未分析。
- 哪些文件被忽略。
- 忽略或缺失文件是否可能影响结论完整性。

日志白名单需要支持 `scope_strategy`：

- `current_snapshot_only`：该日志只属于当前小包对应的启动。
- `multi_boot_split`：该日志包含最近多次启动，需要按启动标志切分。
- `rolling_log_extract`：滚动日志，尽量提取与当前启动相关的片段。
- `unknown_scope`：范围不确定，只能作为弱证据。

示例配置：

```yaml
id: kernel_history_log
name: Kernel retained boot log
path_patterns:
  - "**/kernel_history*.log"
  - "**/dmesg*.log"
source_type: kernel
parser: plain_text_log
scope_strategy: multi_boot_split
boot_split_markers:
  - "Linux version"
  - "Booting Linux"
  - "Kernel command line"
max_boot_sessions: 5
max_size_mb: 20
extractors:
  - boot_stage
  - error_keywords
  - driver_probe_events
  - bus_errors
related_playbooks:
  - kernel_boot_abnormal
  - driver_probe_failure
  - i2c_spi_mdio_bus_error
```

## 10. 启动会话模型

启动过程重建后，系统形成 `BootSession`：

```text
BootSession
- session_id
- snapshot_index
- display_name
- source_archives
- bootloader_logs
- kernel_logs
- userspace_logs
- board_service_logs
- config_and_version_evidence
- key_events
- final_state
- abnormal_stage
- reset_detected
- confidence
- strong_evidence
- weak_evidence
- missing_information
```

启动概览需要展示最近 N 次启动：

```text
最近第 1 次：Kernel 阶段复位，未进入用户态，疑似 watchdog/reset
最近第 2 次：启动成功，用户态板级驱动初始化完成
最近第 3 次：用户态板级驱动启动失败，系统未复位
最近第 4 次：Bootloader 阶段外设初始化失败后复位
```

## 11. 大日志处理策略

单个日志文件可能非常大，系统不能直接把完整日志交给大模型，也不能要求工程师提前人工判断所有有用信息。MVP 采用“规则切片 + 候选片段 + AI 事实抽取 + 诊断模板约束”的处理方式。

处理流程：

```text
大日志文件
  -> 按白名单选择可分析日志
  -> 按日志类型执行规则切片
  -> 提取候选片段
  -> 对候选片段打标签和排序
  -> AI 对高价值片段做事实抽取和重要性判断
  -> 诊断模板约束 AI 分析方向
```

### 11.1 规则切片

每种白名单日志可以配置切片策略。常见策略包括：

- 按启动边界切片，例如 `U-Boot` banner、`Linux version`、`Booting Linux`。
- 按阶段切片，例如 bootloader、kernel、userspace、board service。
- 按错误关键词附近切片，例如 `error`、`fail`、`timeout`、`reset`、`watchdog`、`probe failed`。
- 按模块名或设备名切片，例如驱动名、总线名、服务名、设备名。
- 按上下文窗口切片，例如命中行前后各 50 行或 200 行。
- 按日志尾部切片，例如复位、崩溃、卡住问题优先保留最后若干行。

规则切片由代码和配置完成，不能依赖大模型自由读取完整日志。

### 11.2 候选片段分级

切片后的候选片段需要打标签和排序，决定哪些片段进入 AI 分析。候选片段标签包括：

- 启动阶段片段。
- 错误或告警片段。
- 复位或 watchdog 片段。
- 驱动 probe 片段。
- 升级、启动区、回退片段。
- 用户态服务启动片段。
- 版本、板卡、芯片信息片段。
- 普通噪声片段。

高价值片段优先送入模型。普通噪声默认不送入模型，除非后续追问或诊断模板需要补充检索。

### 11.3 AI 事实抽取

AI 主要用于理解候选片段，输出结构化事实，而不是全量扫描日志。

示例输出：

```json
{
  "stage": "kernel",
  "event_type": "driver_probe_failure",
  "module": "xxx_driver",
  "error_code": "-110",
  "evidence": "probe failed with error -110",
  "possible_meaning": "device access timeout",
  "confidence": "medium"
}
```

事实抽取结果必须保留证据片段，便于后续报告追溯。

### 11.4 诊断模板约束

诊断模板决定“应该怎么看日志”，而不是简单逐行匹配。模板需要声明：

- 关注哪些启动阶段。
- 关注哪些日志类型。
- 必须抽取哪些字段。
- 哪些现象不能过早下结论。
- 常见根因有哪些。
- 需要补充哪些证据。

例如启动区域异常诊断模板应关注：

- bootloader 实际选择的启动区。
- 用户或升级流程期望的启动区。
- 升级成功或失败标志。
- 镜像校验结果。
- 自动回退证据。
- reset 或 watchdog 证据。
- kernel cmdline 中体现的实际启动区。

## 12. 诊断模板

诊断模板用来沉淀板级工程师的人工经验。MVP 使用 Git/文件方式维护，后续再考虑网页后台。

建议目录：

```text
configs/
  log_whitelist/
  diagnosis_playbooks/
```

诊断模板建议包含：

- 模板 ID、名称、版本、状态。
- 适用日志来源。
- 必需证据和可选证据。
- 分析步骤。
- 常见根因。
- 常见误判。
- 输出要求。
- 关联白名单日志类型。

示例：

```yaml
id: driver_probe_failure
name: Linux 驱动 probe 失败诊断
version: 1.0.0
status: active
applicable_sources:
  - kernel
  - device_tree
  - board_config
required_evidence:
  - driver name
  - probe error code
  - device tree node
  - bus access result
analysis_steps:
  - 判断失败发生在哪次启动、哪个阶段
  - 提取驱动名、设备节点、错误码和总线访问结果
  - 检查 power、reset、clock、pinmux、地址、compatible 是否异常
  - 检索历史案例和提交记录作为辅助证据
output_requirements:
  - 直接证据
  - 疑似根因排序
  - 下一步补充信息
```

## 13. 报告结构

分析结果页和报告建议包含四个区域。

### 13.1 针对用户问题的直接回答

放在最上方，优先回答用户最关心的问题。

内容包括：

- 直接回答。
- 当前证据是否充分。
- 最可能原因。
- 不能确认的点。
- 建议下一步。

### 13.2 最近启动概览

多次启动日志包展示最近 N 次启动概览。单次启动日志包展示当前上传启动的详细过程，并提示是否包含历史启动片段。

示例：

```text
序号 | 启动状态 | 异常阶段 | 是否复位 | 关键现象 | 证据文件
0    | 异常     | bootloader/kernel | 是 | 未进入预期启动区 | boot.log
1    | 成功     | -                 | 否 | 进入用户态        | startup.log
```

每次启动可展开阶段时间线：

```text
bootloader -> kernel -> userspace -> board service
```

### 13.3 主动诊断发现

展示第二层诊断模板的结果：

- 诊断模板名称。
- 关联启动会话。
- 诊断发现。
- 关键证据。
- 置信度。
- 建议下一步排查。

### 13.4 证据与分析范围

报告必须明确区分：

- 当前日志直接证据。
- 启动重建事实。
- 诊断模板推理。
- 本地已审核案例辅助证据。
- 历史问题单或代码提交辅助证据。
- 弱证据和不确定假设。

## 14. 追问问答

用户可以在报告生成后继续追问。

追问回答必须基于：

- 原始上传日志包。
- 已分析的白名单日志。
- 启动过程重建结果。
- 诊断模板主动分析结果。
- 已生成报告。
- 本地已审核问答案例和其他辅助知识。

如果当前日志不足以支撑回答，系统需要明确说明不足，并给出需要补充的信息。

## 15. 后台审核与知识沉淀

每一次分析任务都需要完整留痕，但只有工程师确认有效的结果才能进入可复用知识集。

后台诊断详情需要展示：

- 用户原始问题。
- 上传包元信息。
- 输入包类型和启动快照数量。
- 白名单命中文件和忽略文件。
- 启动过程重建结果。
- 主动诊断模板结果。
- 针对用户问题的最终回答。
- 关键日志证据。
- 使用的模型、提示词版本、诊断模板版本。
- 用户追问记录。
- 用户或工程师反馈。
- 是否已入库。

工程师可以执行：

- 点赞或标记本次诊断有效。
- 编辑入库摘要。
- 确认加入本地问答知识集。
- 对已入库案例进行启用或停用。

入库案例结构：

```text
ApprovedDiagnosisCase
- original_question
- problem_tags
- board_model
- chip_model
- software_version
- input_log_summary
- boot_reconstruction_summary
- active_diagnosis_summary
- final_effective_conclusion
- key_evidence_fragments
- diagnosis_process
- solution_or_next_action
- applicable_conditions
- non_applicable_conditions
- source_analysis_task_id
- reviewer
- approved_at
- status
```

闭环如下：

```text
人工经验 -> 诊断模板
当前日志 -> 三层分析
有效结果 -> 工程师确认
确认案例 -> 本地问答知识集
后续诊断 -> 已审核案例作为辅助证据
```

## 16. 知识来源

系统需要支持以下知识来源：

- 工程师维护的诊断模板。
- 日志白名单配置。
- 本地已审核问答案例。
- 内部问题单系统。
- 代码提交记录。
- 人工整理的典型案例。

MVP 优先级：

- 诊断模板。
- 日志白名单。
- 本地已审核问答案例。
- 内部问题单系统。
- 代码提交记录。
- 人工典型案例。

历史问题单、提交记录和典型案例只能作为辅助证据，不能覆盖当前日志直接证据。

## 17. AI 使用边界与系统形态

MVP 应采用“确定性工作流 + 受控大模型调用”的系统形态，不建议第一版做成完全自由 Agent。

### 17.1 不使用 AI 的环节

以下环节应由确定性代码、规则或配置完成：

- 压缩包解压和安全检查。
- 大包和小包识别。
- 小包尾缀 `0/1/2` 的启动快照排序。
- 白名单日志匹配。
- 文件分类：已分析、识别但未分析、忽略。
- 文件大小、编码、路径安全检查。
- 诊断模板版本管理。
- 工程师确认入库。

这些环节需要稳定、可审计，不能由模型自由判断。

### 17.2 使用 AI 的环节

以下环节适合使用大模型能力：

- 非结构化日志候选片段的事实抽取。
- 启动过程重建中的模糊判断，例如是否进入用户态、是否像 watchdog 复位。
- 按诊断模板执行分析步骤。
- 多个诊断结果之间的综合判断。
- 针对用户具体问题的动态回答。
- 后续追问问答。
- 工程师确认入库前的案例摘要整理。
- 历史问题单、提交记录、典型案例的摘要、向量化和重排序。

### 17.3 Agent 与工作流选择

第一阶段采用固定工作流：

```text
上传日志
  -> 规则解压、识别、白名单匹配
  -> 规则切片和候选片段提取
  -> AI 事实抽取
  -> 启动过程重建
  -> AI 按诊断模板分析
  -> AI 综合回答用户问题
  -> 报告生成
  -> 人工审核入库
```

后续可以在第三层动态回答和追问模块中引入受控 Agent。该 Agent 只能使用受限工具：

- 读取本次分析任务数据。
- 读取启动重建结果。
- 读取诊断模板结果。
- 检索白名单日志片段。
- 检索本地已审核案例。
- 检索内部问题单和代码提交辅助证据。

该 Agent 不能：

- 执行系统命令。
- 远程登录板卡。
- 修改诊断模板。
- 自动确认入库。
- 绕过白名单读取未授权文件。
- 将日志发送到非受控外部服务。

## 18. 模型要求

第一版接入 MiniMax M2.7 API。

系统需要提供模型适配层，业务逻辑不直接绑定具体模型。

内部接口建议抽象为：

- `chat`
- `analyze`
- `summarize`
- `embed`

后续可以切换到 Qwen、DeepSeek、Llama 或公司统一大模型服务。

模型输出尽量采用结构化 JSON，特别是启动事实、诊断发现、报告段落等结果。

## 19. 非功能要求

### 19.1 安全

- 内网私有化部署。
- 日志不得离开公司受控环境。
- 压缩包解压需要安全检查。
- 模型请求和响应需要记录元信息，便于审计。
- 不向无关用户暴露敏感日志。

### 19.2 可追溯

每个结论都应尽量追溯到：

- 来源文件。
- 日志片段。
- 启动会话。
- 诊断模板。
- 模型调用。
- 已审核历史案例。

### 19.3 可扩展

系统后续应支持扩展：

- 新日志白名单。
- 新解析器。
- 新启动切分策略。
- 新诊断模板。
- 新模型供应商。
- 新知识源连接器。

### 19.4 可靠性

系统需要优雅降级：

- 部分文件解析失败时，继续基于可用文件分析。
- 日志归属不确定时，标记为弱证据。
- 模型输出不完整时，允许重试或展示部分结果。
- 历史知识检索失败时，仍然输出基于当前日志的诊断。

## 20. MVP 功能范围

MVP 包含：

- 分析任务创建。
- 用户问题输入。
- 日志压缩包上传。
- 多次启动包和单次启动包自动识别。
- 压缩包解压和文件清单生成。
- 白名单日志匹配。
- 基础文本日志解析和关键事件抽取。
- 启动过程重建。
- 诊断模板主动分析。
- 用户问题动态回答。
- 分析报告页面。
- 追问问答。
- 后台诊断任务列表。
- 后台诊断详情页。
- 工程师确认有效并入库。
- 已审核案例检索和引用。
- MiniMax M2.7 模型适配。

MVP 不包含：

- 网页化诊断模板编辑器。
- 复杂权限模型。
- 自动板卡命令执行。
- 完整问题单流转。
- AI 自动修改诊断模板。
- 压缩包全文件无差别分析。

## 21. 验收标准

- 给定一个包含尾缀 `0`、`1`、`2` 小压缩包的多次启动日志包，系统能按正确顺序识别三次启动快照。
- 给定一个单次启动小压缩包，系统能将它作为 `snapshot_index = 0` 分析。
- 给定白名单和非白名单文件，系统只分析白名单文件，并在报告中列出忽略文件。
- 给定包含多次启动记录的 kernel 保留日志，系统能按 `scope_strategy` 切分或标记片段归属。
- 给定重建后的启动会话，报告能展示每次启动状态、异常阶段、是否复位和证据文件。
- 给定已配置诊断模板，系统能运行相关模板并输出结构化诊断发现。
- 给定用户问题，报告第一部分能给出基于启动重建和诊断结果的直接回答。
- 当证据不足时，系统能明确说明不能得出的结论和需要补充的信息。
- 工程师能在后台查看每次诊断详情，并将有效结果确认加入本地问答知识集。
- 后续分析能检索已审核案例，并在报告中明确标注其为历史辅助证据。
- 给定一个大日志文件，系统能先通过规则切片提取候选片段，再将高价值片段送入 AI 做事实抽取。
- 给定一个低价值普通日志片段，系统不会默认将其送入模型，除非诊断模板或用户追问需要。
- 给定一个需要动态回答的用户问题，系统能基于前两层结果组织回答，而不是依赖固定问题模板。
