## 未完成或偏弱项

下面是对 PRD 完整闭环的缺口，不影响当前 MVP 计划任务已落地，但影响“产品级完成”判断。

### 数据持久化与运行形态

- 当前 API 工作流主要使用内存型任务存储；SQLAlchemy 表结构已建，但任务运行链路尚未真正落到 SQLite 持久化。
- 服务重启后，任务、追问、审核案例会丢失。
- 上传文件存储存在本地目录配置，但缺少清理、容量治理和长期归档策略。

### 已审核案例复用

- 已支持审核入库和案例列表。
- 尚未实现“后续诊断检索已审核案例，并在报告中标注为历史辅助证据”。
- 尚未实现案例启用/停用 API 与前端操作。
- 尚未实现案例搜索、相似问题匹配、向量检索或重排序。

### 报告完整性

- 结果页展示了核心回答、启动表、诊断发现、证据和追问。
- 尚未完整展示白名单命中文件、忽略文件、识别但未分析文件。
- 尚未完整展示模型调用元信息、提示词版本、诊断模板版本。
- “证据不足时明确说明不能得出的结论和需补充信息”目前只有部分规则与测试覆盖，仍需要更强的报告层表达。

### 模型接入深度

- MiniMax provider 已有配置入口，但真实 API 调用仍是占位，单元测试不会调用外部模型。
- 模型请求/响应审计日志未落库。
- 模型输出结构化校验与失败重试机制还不完整。

### 前端产品化

- 目前是轻量级状态切换，没有正式路由和 URL 可分享能力。
- 缺少任务详情深链、返回按钮、错误恢复、加载骨架等体验细节。
- 进度页是组件级实现，尚未接入真实任务状态轮询。
- 审核页的追问历史、模型版本、模板版本仍是占位或间接展示。

### 后端健壮性

- 当前 task runner 是同步 MVP 形态，没有后台队列、任务取消、失败重试、超时控制。
- 大日志处理只有基础切片和评分，尚未实现更完整的大文件流式处理。
- 文件分类只有 `analyzed` 和 `ignored` 的主路径，`recognized_not_analyzed` 还未形成完整流程。
- 弱证据、不确定归属和缺失证据的结构化表达还需要加强。

### 部署与运维

- 尚未提供 Docker 或内网部署脚本。
- 尚未提供生产配置模板、日志配置、审计配置。
- 尚未提供权限、用户、角色、访问控制。
- 尚未接入 CI。

## 下一步建议

优先级建议：

1. 将 `TaskRunner` 从内存存储迁移到 SQLite 持久化，保证任务、追问、审核案例服务重启后不丢失。
2. 增加已审核案例检索与诊断引用闭环，让入库案例真正参与后续分析。
3. 完善报告中的文件范围、忽略原因、模型/模板版本、证据不足说明。
4. 接入真实 MiniMax API 调用与请求审计日志。
5. 给前端引入正式路由、任务详情深链、真实进度轮询和案例启停操作。
